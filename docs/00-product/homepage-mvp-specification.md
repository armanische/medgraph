## Статус документа и границы решений

После явного продуктового утверждения этот документ становится единственным источником требований к Homepage MVP. Реализация не должна самостоятельно менять порядок блоков, иерархию действий, правила выбора данных, тексты обязательных элементов или поведение деградации.

Любое отступление оформляется как изменение этой спецификации до изменения дизайна или кода. Документ заменяет требования к Homepage MVP из `docs/homepage-information-architecture.md`; прежний RFC остаётся историческим описанием ранее реализованного состояния, но не определяет новый MVP.

Спецификация определяет продукт, контент, информационную архитектуру и поведение. Она не определяет визуальный стиль, сетку, цвета, изображения, конкретные компоненты или техническую реализацию. Эти решения относятся к Homepage MVP Design v1 и последующей implementation-задаче, но обязаны соблюдать зафиксированный здесь контракт.

В scope входят:

- роль Homepage в основном пользовательском пути;
- последовательность и назначение блоков;
- обязательный публичный контент;
- переходы в существующие сценарии;
- responsive-поведение на уровне структуры;
- измеримые критерии успеха и приёмки.

В scope не входят:

- изменение Product Detail, Catalog Experience или Structured Fields;
- новый поиск, новый каталог, новые публичные маршруты или новая data model;
- персонализация, рекомендации, сравнение, избранное и авторизация;
- изменение Product Data, публикации, review state или Cloud Catalog;
- обещания поставки, партнёрства или экспертизы без отдельно принятого доказательства;
- дизайн, Figma, HTML, React, Tailwind, компоненты, изображения и реализация.

## 1. Purpose

Homepage — не рекламный лендинг и не сокращённая копия каталога. Это входная точка в каталог медицинского оборудования, которая помогает пользователю быстро выбрать подходящий способ поиска и продолжить существующий сценарий.

Основной путь:

```text
Homepage
  → Catalog
  → Product Detail
  → Request for quotation
```

Homepage решает одну основную задачу: **сокращает время от входа на сайт до начала осмысленного поиска оборудования**.

Страница должна ответить пользователю на четыре вопроса:

1. Что находится в CyberMedica?
2. Как найти оборудование по известному названию, модели, категории или производителю?
3. Какие данные можно увидеть перед отправкой запроса?
4. Что делать, если нужная модель не найдена или требуется помощь с выбором?

Пользователь должен продолжить путь, потому что Homepage:

- сразу даёт поиск и прямой переход в каталог;
- предлагает понятные входы по категории и производителю;
- объясняет практическую пользу сервиса без абстрактных обещаний;
- оставляет запрос специалисту как безопасный запасной путь.

Homepage считается успешной не тогда, когда пользователь долго остаётся на ней, а когда он быстро переходит к релевантному каталожному действию.

## 2. User Goals

### 2.1. Основные пользовательские намерения

| Намерение | Что уже знает пользователь | Необходимый путь с Homepage | Успешный результат |
| --- | --- | --- | --- |
| Найти конкретное изделие | Название, модель или производитель | Hero Search → Catalog results → Product Detail | Открыта релевантная карточка |
| Выбрать тип оборудования | Задача или категория, но не модель | Featured Categories → filtered Catalog → Product Detail | Начат просмотр подходящей категории |
| Посмотреть оборудование бренда | Производитель | Manufacturers → Manufacturer page → Product Detail | Открыт ассортимент производителя |
| Сравнить доступные варианты вручную | Класс оборудования и несколько требований | Catalog → Product Detail pages | Пользователь получил данные для оценки |
| Получить помощь | Нужного товара нет или запрос неоднозначен | Final CTA → Request | Отправлен осмысленный запрос |

### 2.2. Приоритет пользователей

Homepage проектируется прежде всего для профессионального B2B-поиска:

- медицинских организаций и клиник;
- специалистов по закупкам;
- руководителей и сотрудников профильных подразделений;
- пользователей, которым необходимо найти изделие, документацию или возможный аналог.

Homepage не должна предполагать, что пользователь знает внутреннюю структуру каталога, профессиональную терминологию CyberMedica или точный артикул.

### 2.3. Пользовательские вопросы, которые страница обязана закрыть

- Можно ли искать по названию, модели, производителю или категории?
- Можно ли перейти к полному каталогу без промежуточного шага?
- Есть ли нужная категория оборудования?
- Представлен ли интересующий производитель?
- Какие сведения доступны в карточке товара?
- Куда обратиться, если результат не найден или нужен аналог?

Страница не обязана отвечать на вопросы о цене, сроке поставки, наличии, совместимости или регуляторном статусе конкретного изделия. Такие сведения нельзя обещать без достоверных данных и соответствующего продуктового процесса.

## 3. Homepage Principles

### 3.1. Catalog first

Каждый основной блок должен либо вести в каталог, либо помогать выбрать каталоговый путь. Контент без такого назначения в Homepage MVP не включается.

### 3.2. Search first, but not search only

Пользователь с конкретным запросом получает поиск на первом экране. Пользователь без точного запроса получает альтернативный вход через категории и производителей.

### 3.3. One primary journey

Переход в каталог — основное действие страницы. Запрос специалисту — вторичное действие для случаев, когда самостоятельный поиск не решил задачу.

### 3.4. No unsupported claims

Homepage не использует формулировки «официальные поставки», «официальный партнёр», «полный каталог», «проверено», «лучший», «ведущий» или аналогичные утверждения без отдельного документального и юридического подтверждения.

### 3.5. Real public data only

Категории и производители берутся из того же публичного Storefront-контракта и активного источника данных, что и Catalog Experience. Нельзя создавать локальные списки, фиктивные сущности, жёстко заданные counts или специальные homepage-only product records.

### 3.6. Fail closed

Отсутствующие сведения скрываются. Они не заменяются предположениями, маркетинговым текстом или внутренними статусами. Ошибка одного динамического блока не должна превращать Homepage в недостоверную витрину.

### 3.7. No catalog duplication

Homepage показывает направления входа, а не товарную выдачу. В MVP не размещается сетка популярных, рекомендуемых или новых товаров.

### 3.8. Stable hierarchy across viewports

Desktop, tablet и mobile содержат один и тот же смысл, порядок блоков и приоритет CTA. Responsive-поведение меняет компоновку, но не продуктовую иерархию.

### 3.9. Public terminology only

На странице запрещены internal IDs, source UID, checksums, review/publication terminology, quality flags, preview/staging wording и технические названия источников данных.

### 3.10. Existing launch baseline remains authoritative

Homepage использует существующие маршруты и публичные сервисы. Она не изменяет контракты Product Detail, Catalog Experience или Structured Fields и не вводит параллельный runtime-путь.

## 4. Information Architecture

### 4.1. Утверждённая последовательность

После существующего глобального Header Homepage содержит ровно пять продуктовых блоков:

1. **Hero with Search**
2. **Featured Categories**
3. **Manufacturers**
4. **Why CyberMedica**
5. **Final CTA**

Footer остаётся глобальным элементом и не определяется этой спецификацией.

### 4.2. Назначение блоков

| Порядок | Блок | Цель | Содержание | Основное действие | Критерий успеха |
| --- | --- | --- | --- | --- | --- |
| 1 | Hero with Search | Сразу объяснить продукт и начать поиск | H1, пояснение, поиск, ссылка в каталог | Найти оборудование | Search submit или переход в Catalog |
| 2 | Featured Categories | Дать путь пользователю без конкретной модели | До 6 доступных категорий | Открыть категорию в Catalog | Category click |
| 3 | Manufacturers | Поддержать бренд-ориентированный поиск | До 8 производителей с товарами | Открыть производителя | Manufacturer click |
| 4 | Why CyberMedica | Объяснить практическую пользу продолжения | 4 подтверждаемых возможности сервиса | Не требует отдельного CTA | Переход к следующему действию без ухода со страницы |
| 5 | Final CTA | Завершить страницу ясным выбором | Краткая помощь, Catalog CTA, Request CTA | Перейти в Catalog | Catalog или Request transition |

### 4.3. Что намеренно исключено из MVP

**Popular Products.** Блок исключён: в Launch Baseline нет утверждённого аналитического или редакционного сигнала «популярности». Случайная, статическая или техническая выборка создаёт недостоверную иерархию и дублирует Catalog.

**Отдельный блок Application Areas.** Области применения остаются способом уточнения внутри Catalog. Одновременный показ категорий, производителей и областей применения на Homepage создаёт избыточный выбор и смешивает разные таксономии.

**Статистическая сводка.** Counts товаров, категорий и производителей не являются самостоятельной пользовательской ценностью и придают странице вид административной панели. Динамический count допустим только как вспомогательное значение внутри конкретной категории или производителя.

**Отзывы, новости, карусели, логотипы клиентов, награды и неподтверждённые trust claims.** Эти элементы не сокращают основной путь и требуют отдельных доказательств или контентного процесса.

### 4.4. Навигационный контракт

- Hero Search передаёт запрос в существующий Catalog через URL.
- Категория открывает Catalog с соответствующим публичным category-параметром.
- Производитель открывает существующую каноническую страницу производителя.
- Общий Catalog CTA ведёт на `/catalog`.
- Request CTA ведёт на существующий маршрут `/request`.
- Homepage не создаёт новые промежуточные страницы.
- Возврат из Product Detail в Catalog продолжает использовать действующий recovery-контракт Catalog Experience; Homepage его не заменяет.

## 5. Hero

### 5.1. Цель

Hero должен за один просмотр объяснить, что CyberMedica — каталог медицинского оборудования для профессионального выбора и запроса, а затем предложить немедленное действие.

Hero отвечает на вопросы:

- **Что это?** Каталог медицинского оборудования.
- **Для кого?** Для клиник, медицинских организаций и специалистов по закупкам.
- **Почему продолжить?** Можно найти изделие и открыть доступные характеристики и документы перед запросом.

### 5.2. Обязательный контент

**H1**

> Медицинское оборудование для клиник и медицинских организаций

**Supporting copy**

> Найдите оборудование по названию, модели, производителю или категории и перейдите к карточке с доступными характеристиками и документами.

**Search label**

> Найти оборудование

**Search placeholder**

> Название, модель, производитель или категория

**Search submit**

> Найти

**Secondary action**

> Перейти в каталог

### 5.3. Поведение поиска

- Поле и submit образуют семантическую search form.
- Непустой запрос открывает `/catalog` с существующим query-параметром `q`.
- Значение сохраняется в URL, чтобы результат можно было обновить, скопировать и восстановить после Product Detail.
- Пустой submit не меняет маршрут и не показывает ошибку; фокус остаётся в поле.
- Поиск не обещает autocomplete, исправление опечаток или интеллектуальные рекомендации, если это не поддерживает существующий Catalog Experience.
- Homepage не создаёт отдельный search endpoint или собственный поисковый индекс.

### 5.4. CTA hierarchy

1. Search submit — первичное действие Hero.
2. `Перейти в каталог` — вторичное действие Hero.
3. Request for quotation в Hero не показывается: запрос до выбора оборудования преждевременно конкурирует с основным сценарием.

### 5.5. Ограничения

- Один H1 на странице.
- Без слоганов, превосходных степеней и неподтверждённых обещаний.
- Без карусели, видео, product hero card и декоративных элементов, которые добавляют новый смысл или CTA.
- Без hardcoded каталожных counts.
- Без повторения Header navigation внутри Hero.

## 6. Catalog Entry

Homepage использует комбинацию трёх входов в строгом порядке:

1. **Поиск** — для пользователя с известным названием, моделью, производителем или категорией.
2. **Категории** — для пользователя, который знает тип оборудования, но не конкретное изделие.
3. **Производители** — для пользователя с предпочтением или закупочным требованием по бренду.

Категории показываются раньше производителей, потому что большинство исследовательских сценариев начинается с класса оборудования. Пользователь с конкретным брендом уже может использовать Hero Search, но получает дополнительный обзорный путь следующим блоком.

Homepage не показывает товарную сетку. Прямой переход в Product Detail возникает из результатов Catalog или со страницы производителя. Это сохраняет единую выдачу, единые правила состояния и предсказуемый Back-to-Catalog flow.

### 6.1. Data contract

- Все динамические сущности загружаются через существующие Storefront Services и `CatalogRepository`.
- Источник данных совпадает с активным источником Catalog в текущей среде.
- Homepage не читает Supabase, RPC, local JSON или immutable data напрямую.
- Для блоков используются только list-level public fields; Product Detail payload для всех товаров не загружается.
- Нельзя создавать N+1-запросы ради counts, logos или описаний.
- В public output не попадают unpublished/internal fields.

### 6.2. Degraded state

- Hero, Search и общий Catalog CTA остаются доступными, даже если один обзорный блок не имеет валидных данных.
- Пустой динамический блок полностью скрывается вместе с заголовком; пустые карточки и fabricated fallback entities запрещены.
- При recoverable data-source error пользователь получает спокойное публичное сообщение и путь в Catalog, если это поддерживает текущая error boundary.
- Runtime error, stack trace, internal endpoint или environment detail пользователю не показываются.

## 7. Featured Categories

### 7.1. Решение

Блок категорий обязателен при наличии валидных публичных категорий с товарами. Он является первым обзорным блоком после Hero.

**Название блока**

> Категории оборудования

**Общий CTA**

> Все категории

CTA ведёт на `/catalog` без фильтра.

### 7.2. Содержание

Показывается не более 6 категорий. Каждая entry содержит:

- публичное название категории;
- краткое публичное описание, только если оно доступно;
- актуальное количество доступных товаров, только если оно уже получено без дополнительного per-item запроса;
- ссылку в Catalog с существующим category-параметром.

Изображение, иконка или описание не создаются автоматически из названия категории. Визуальный дизайн без дополнительных данных определяется в Homepage MVP Design v1.

### 7.3. Правило выбора

В MVP используются только активные категории, в которых есть доступные Storefront products. Порядок воспроизводим:

1. количество доступных товаров по убыванию;
2. публичное название по возрастанию при равном количестве.

Это правило отражает наполненность каталога, но не называет категории «популярными». Ручная editorial curation и персонализация не входят в MVP.

### 7.4. Success criteria

- пользователь понимает основные классы доступного оборудования;
- category click открывает Catalog с применённым фильтром;
- URL однозначно отражает выбранную категорию;
- при возврате из Product Detail действующий Catalog recovery сохраняет состояние.

## 8. Manufacturers

### 8.1. Решение

Блок производителей нужен: в B2B-закупке бренд может быть исходным требованием, а переход к странице производителя сокращает путь к его ассортименту.

**Название блока**

> Производители

**Общий CTA**

> Все производители

CTA ведёт на существующий каталог производителей.

### 8.2. Содержание и формат

Показывается не более 8 производителей. Каждая entry содержит:

- каноническое публичное название;
- страну, только если она заполнена и публично безопасна;
- количество доступных товаров, если оно уже рассчитано без N+1;
- существующий подтверждённый logo asset, если он доступен;
- ссылку на каноническую страницу производителя.

При отсутствии логотипа используется текстовая идентичность. Нельзя подменять логотип placeholder-брендом, извлекать его из внешней сети во время runtime или интерпретировать наличие производителя как официальное партнёрство CyberMedica.

### 8.3. Правило выбора

В блок входят только активные производители минимум с одним доступным Storefront product. Порядок воспроизводим:

1. количество доступных товаров по убыванию;
2. публичное название по возрастанию при равном количестве.

### 8.4. Польза

- быстрый доступ к ассортименту известного бренда;
- дополнительный путь обнаружения оборудования;
- укрепление доверия через прозрачное указание изготовителя, а не через неподтверждённые partner claims.

## 9. Why CyberMedica

### 9.1. Цель

Блок объясняет практические возможности сервиса после того, как пользователь увидел способы входа в Catalog. Он не должен выглядеть как набор рекламных обещаний.

**Название блока**

> Как CyberMedica помогает с выбором

### 9.2. Обязательные смысловые пункты

1. **Подбор под задачу** — поиск по каталогу и навигация по категории и производителю помогают сузить выбор оборудования.

2. **Помощь с поиском аналогов** — если нужная модель не найдена, пользователь может отправить запрос и описать требования. CyberMedica не заявляет эквивалентность изделий без проверки.

3. **Доступные характеристики и документы** — карточка показывает только имеющиеся публичные характеристики, регистрационные сведения и файлы; полнота не обещается.

4. **Сопровождение запроса** — специалист помогает уточнить модель, комплектацию и исходные требования к запросу.

Каждый пункт выражает одну короткую проверяемую мысль. Дополнительный CTA внутри блока не нужен: следующий блок содержит завершение сценария.

### 9.3. Claims policy

Формулировки об официальных поставках, статусе дилера, опыте государственных закупок, сроках ответа, наличии, цене или гарантиях не входят в MVP без отдельного evidence package и продуктового утверждения. Их отсутствие нельзя компенсировать близкой по смыслу рекламной формулировкой.

## 10. Final CTA

### 10.1. Цель

Final CTA даёт следующий шаг пользователю, который просмотрел Homepage, но ещё не начал поиск или не уверен, что найдёт нужное изделие самостоятельно.

**Heading**

> Не нашли нужную модель?

**Supporting copy**

> Перейдите в полный каталог или отправьте запрос — специалист поможет уточнить модель, комплектацию или возможный аналог.

**Primary action**

> Перейти в каталог

Переход: `/catalog`.

**Secondary action**

> Отправить запрос

Переход: `/request`.

### 10.2. Правила

- Catalog остаётся первым и главным действием.
- Request не должен выглядеть обязательным условием доступа к каталогу.
- Не обещаются конкретное время ответа, наличие, цена или поставка.
- В Homepage не размещается полная RFQ-форма; форма остаётся на существующем dedicated route.
- Дополнительные CTA, телефонные pop-up, sticky request button и сравнение в MVP не добавляются.

## 11. Mobile-first Behaviour

### 11.1. Общие правила

- Порядок блоков и CTA одинаков на всех viewport.
- Контент адаптируется reflow, а не дублируется разными DOM-версиями.
- Интерактивные элементы доступны с клавиатуры и имеют различимое focus state.
- Tap target должен быть не менее 44 × 44 CSS px.
- Горизонтальный scroll страницы запрещён.
- Карусели и скрытые swipe-only списки не используются.
- Search label сохраняется как доступное имя независимо от визуального решения.
- Семантический порядок heading: один H1, затем H2 блоков, без пропуска уровней.

### 11.2. Desktop

- Hero содержит пояснение, Search и оба способа входа без изменения их приоритета.
- Категории, производители и преимущества могут использовать многоколоночную компоновку.
- Все доступные элементы выбранной MVP-выборки видимы без carousel interaction.
- Final CTA показывает Catalog action первым, Request — вторым.

### 11.3. Tablet

- Сохраняется desktop-порядок, но обзорные списки переходят к меньшему числу колонок.
- Search остаётся самостоятельным первым действием Hero.
- CTA не сливаются в один неоднозначный элемент и не меняются местами.
- Не появляется горизонтальная прокрутка карточек.

### 11.4. Mobile

- Структура становится одноколоночной.
- H1 и supporting copy идут перед Search; Search — перед ссылкой в Catalog.
- Категории и производители отображаются как последовательные доступные списки, а не горизонтальные карусели.
- Внутри Final CTA действие `Перейти в каталог` расположено раньше `Отправить запрос`.
- Ни один основной путь не скрывается в accordion, swipe, hover или desktop-only navigation.
- Homepage не вводит sticky RFQ action.

### 11.5. Content parity

Mobile не получает сокращённую продуктовую версию страницы. Допустимо скрывать только необязательные описания или counts по решению Design v1, если название сущности, ссылка и смысл блока остаются доступны. Нельзя скрывать Search, общий Catalog CTA, Featured Categories, Manufacturers или Final CTA при наличии валидных данных.

## 12. Homepage KPIs

### 12.1. Primary KPIs

| KPI | Определение | Что подтверждает |
| --- | --- | --- |
| Catalog entry rate | Доля сессий Homepage с переходом в `/catalog`, включая Search и category entry | Homepage приводит пользователя к каталогу |
| Search submit rate | Доля сессий Homepage с непустым search submit | Первый экран поддерживает known-item поиск |
| Product discovery rate | Доля сессий, начавшихся на Homepage и дошедших до Product Detail | Пользователь находит оборудование, а не останавливается на навигации |
| RFQ completion after Homepage | Доля сессий Homepage с завершённой отправкой существующего Request/RFQ flow | Основной коммерческий путь завершается действием |

### 12.2. Secondary KPIs

- category entry click-through rate;
- manufacturer entry click-through rate;
- переход в общий каталог из Hero и Final CTA по отдельности;
- Request CTA click-through и completion rate;
- доля search zero-result sessions;
- медианное время до первого catalog-related action;
- Product Detail views после category и manufacturer entry.

### 12.3. Guardrail metrics

- runtime error rate Homepage;
- доля неуспешных data loads динамических блоков;
- отсутствие horizontal overflow на поддерживаемых viewport;
- отсутствие blocking accessibility findings;
- отсутствие ухудшения Catalog/Product Detail cross-flow;
- отсутствие попадания internal fields или unpublished data в public response.

### 12.4. Measurement contract

Предлагаемые смысловые события:

- `homepage_search_submit`;
- `homepage_catalog_open`;
- `homepage_category_open`;
- `homepage_manufacturer_open`;
- `homepage_request_open`;
- `product_view_after_homepage`;
- `rfq_submit_after_homepage`.

События должны содержать только необходимый публичный контекст: placement, публичный slug и route. Полный поисковый запрос, контактные данные, internal IDs и чувствительные данные в analytics payload не передаются.

Если в Launch Baseline нет утверждённой analytics-инфраструктуры, её добавление не входит в Homepage MVP автоматически. KPI остаются measurement contract; подключение нового provider требует отдельной задачи и privacy review.

## 13. Acceptance Criteria

### 13.1. Product and information architecture

- [ ] Homepage выполняет роль входа в каталог, а не рекламного лендинга.
- [ ] Основной путь `Homepage → Catalog → Product Detail → Request` доступен без тупиков.
- [ ] Блоки расположены строго в порядке: Hero with Search, Featured Categories, Manufacturers, Why CyberMedica, Final CTA.
- [ ] На Homepage нет Popular Products, отдельного Application Areas block и самостоятельной статистической сводки.
- [ ] Переход в Catalog является главным действием; Request — вторичным.
- [ ] Страница не дублирует Catalog product grid.

### 13.2. Hero and navigation

- [ ] На странице один H1 с утверждённым смыслом.
- [ ] Hero объясняет продукт, аудиторию и следующий шаг без маркетингового слогана.
- [ ] Search принимает название, модель, производителя или категорию и передаёт непустой запрос в существующий `/catalog` через URL.
- [ ] Пустой submit не меняет маршрут и не создаёт ошибку.
- [ ] Hero содержит прямой переход в `/catalog`.
- [ ] Final CTA ведёт в `/catalog` и `/request` в утверждённом приоритете.
- [ ] Все переходы используют существующие канонические маршруты.

### 13.3. Dynamic content

- [ ] Featured Categories показывает не более 6 активных категорий с доступными товарами по воспроизводимому правилу.
- [ ] Manufacturers показывает не более 8 активных производителей с доступными товарами по воспроизводимому правилу.
- [ ] Названия, slugs, country, counts и logos не создаются и не исправляются presentation-слоем.
- [ ] Пустой блок скрывается полностью; фиктивные fallback entities отсутствуют.
- [ ] Homepage не делает Product Detail fetch для каждой entity и не создаёт N+1.

### 13.4. Data and architecture invariants

- [ ] Все каталожные данные проходят через существующие Storefront Services и `CatalogRepository`.
- [ ] Homepage использует тот же активный data source, что и Catalog.
- [ ] Нет прямого чтения Supabase/RPC, local JSON, snapshot, review или publication artifacts.
- [ ] ProductService, CatalogRepository, Cloud Domain Model, Structured Fields и Product Detail contracts не меняются ради Homepage.
- [ ] Product Data, review state, publication state, migrations и Cloud Catalog не изменяются.
- [ ] Internal IDs, quality/review terms, checksums и unpublished fields отсутствуют в публичном output.

### 13.5. Content integrity

- [ ] Используются утверждённые обязательные тексты или явно утверждённая новая редакция этой спецификации.
- [ ] Нет заявлений об официальных поставках, партнёрстве, полноте, наличии, цене, сроках или гарантиях без evidence approval.
- [ ] Why CyberMedica содержит ровно четыре практических возможности, описанные в разделе 9.
- [ ] Отсутствующие данные не заменяются догадкой или рекламным filler text.

### 13.6. Responsive and accessibility

- [ ] Смысловой порядок и CTA hierarchy одинаковы на desktop, tablet и mobile.
- [ ] Проверены минимум 1440 px, tablet viewport и 390 px.
- [ ] Горизонтальный overflow отсутствует.
- [ ] Search, links и actions доступны с клавиатуры и имеют доступные имена.
- [ ] Tap targets соответствуют минимуму 44 × 44 CSS px.
- [ ] Нет hover-only, swipe-only или carousel-only доступа к основному контенту.
- [ ] Heading structure содержит один H1 и последовательные H2.

### 13.7. Failure behaviour

- [ ] Hero, Search и общий Catalog CTA остаются доступными при отсутствии данных одного обзорного блока.
- [ ] Ошибка не раскрывает stack trace, endpoint, environment, source или внутренний статус.
- [ ] Страница деградирует честно и fail-closed.
- [ ] Homepage не создаёт regression в Catalog state restoration или Product Detail flow.

### 13.8. KPI readiness

- [ ] Для Primary KPI определены точки перехода и completion.
- [ ] Analytics не получает полный поисковый запрос, контактные данные или internal IDs.
- [ ] Отсутствие analytics provider не расширяет scope Homepage MVP без отдельного решения.

### 13.9. Constitution compliance

- [x] Homepage не дублирует Catalog.
- [x] Homepage не превращается в рекламный лендинг.
- [x] Homepage минимизирует путь до оборудования.
- [x] Homepage поддерживает существующий Launch Baseline.
- [x] Homepage не требует изменения Product Detail.
- [x] Homepage не требует изменения Structured Fields.
- [x] Homepage сохраняет Storefront boundary и fail-closed principle.

### 13.10. Definition of Ready for implementation

Homepage MVP может перейти к Design v1 только после явного утверждения этой спецификации владельцем продукта. Design v1 должен определить визуальное представление без изменения IA, copy contract, CTA hierarchy и data rules. Реализация начинается только после отдельного утверждения Design v1.

## Product decision summary

| Решение | Утверждённый вариант | Почему |
| --- | --- | --- |
| Роль Homepage | Вход в Catalog | Минимизирует путь до оборудования |
| Первый способ поиска | Hero Search | Поддерживает known-item intent без промежуточного шага |
| Первый обзорный блок | Featured Categories | Помогает выбрать класс оборудования без знания модели |
| Второй обзорный блок | Manufacturers | Поддерживает бренд-ориентированные закупки |
| Popular Products | Не входит в MVP | Нет доказуемого сигнала популярности; дублирует Catalog |
| Application Areas block | Не входит в MVP | Остаётся уточняющим фильтром Catalog |
| Основной CTA | Перейти в Catalog | Соответствует главному пользовательскому пути |
| Вторичный CTA | Отправить Request | Запасной путь после самостоятельного поиска |
| Data source | Existing Storefront Services + CatalogRepository | Сохраняет Launch Baseline и единый public contract |
| Следующий этап | Homepage MVP Design v1 | Визуализирует утверждённую спецификацию без новых UX-решений |
